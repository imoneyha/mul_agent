# role-mapping
## 四实例角色映射
| 实例 | 主职责 | 典型任务 |
## 机器人命名与编制（最终口径）
| 实例 | 机器人名称（对外） | 领域定位 | 编制要求 |
|---|---|---|---|
| assistant | **白泽** | 智能助手入口 | 1 名助理（仅监督） + N 名下属 agent |
| security | **武安侯** | 安防救援 | 1 名助理（仅监督） + N 名下属 agent |
| creative | **藏书阁** | 内容文创 | 1 名助理（仅监督） + N 名下属 agent |
| devops | **赛博道长** | 系统研发 | 1 名助理（仅监督） + N 名下属 agent |
### 组织规则（强约束）
1. 每个实例是一个**独立机器人**，仅处理本领域单一需求。
2. 每个实例内：
   - **助理（assistant-of-instance）**负责调度、监督、验收与防摸鱼；
   - **下属 agents**负责执行具体任务，可按需求扩容。
3. 统一执行标准：**7x24 小时待命、全年无休、无需重新唤醒**。
4. 资源策略：token **按需给足**，确保随时可执行任务。
5. 管理目标：不因唤醒/配额不足导致任务延迟。
6. 助理角色边界：只做监督/防摸鱼/督促主 agent 验收，不直接承接执行任务。
|---|---|---|
| assistant | 入口与分流 | 用户对话、任务路由、结果回传 |
| security | 安防救援 | 安全巡检、风险审计、故障应急 |
| creative | 内容文创 | 小说创作、文案写作、内容策划 |
| devops | 系统研发 | 代码开发、运维发布、验证回归 |
## 默认路由建议
1. 用户消息默认进入 `assistant`。
2. 涉及安全/权限/审计 → 转 `security`。
3. 涉及写作/文案/创作 → 转 `creative`。
4. 涉及工程实现/部署/脚本 → 转 `devops`。
## 边界约束
- 避免多实例同时绑定同一对外入口会话。
- 每实例只维护本实例职责相关上下文与运行状态。
- 跨实例协作时由 assistant 负责最终对外回复。
